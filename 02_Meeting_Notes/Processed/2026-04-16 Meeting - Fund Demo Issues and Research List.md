## 会议结论摘要

当前基于现有 Ticket 与 demo，已经明确需要覆盖三条主流程：
- Fund Issuance / Subscription
- Fund Redemption
- Fund Distribution

但在真正进入 demo 开发前，团队已经识别出一批关键缺口。  
这些缺口主要集中在：
- 基金类型边界不清
- 链上与链下职责划分不清
- 合规 / KYC / 白名单机制未完全定稿
- 开放式基金、赎回、分红规则仍有若干产品定义空缺
- 当前 demo scope 与交付时间存在冲突

## 一、已识别问题

### 1. 基金类型范围不清，直接影响 demo 方案

已识别问题：
- CCB 侧目前没有明确指定是封闭式基金、开放式基金还是 ETF 类产品。
- 如果同时覆盖封闭式、开放式、ETF 三类，当前排期无法支撑。
- 现有 Ticket 以主流程为主，但未把“基金类型差异”单独拆成明确的产品范围边界。

影响：
- 影响 Ticket 字段设计
- 影响生命周期流程
- 影响赎回 / 分红 / 结算逻辑
- 影响 demo 交付范围与复杂度

当前倾向：
- 优先准备封闭式基金 + 开放式基金
- ETF 暂不纳入本轮 demo

### 2. 现有 Issuance 流程仍偏 Bond/封闭式思维，开放式基金规则不完整

已识别问题：
- 当前 demo 和 Ticket 更容易映射到“封闭募集 -> 关闭申购 -> 分配份额”的封闭式基金逻辑。
- 开放式基金涉及 `T+1 确权`、周期性开放赎回、每日净值结算等规则，现有 Ticket 还未完整覆盖。
- 开放式基金与 ETF 的差异虽然在讨论中被提到，但尚未形成统一产品定义。

影响：
- 影响 `Fund type` 选择后的字段和状态机
- 影响 `Redemption` 是否长期存在、按窗口开放还是固定日开放
- 影响 `NAV / settlement` 相关页面和后台设计

### 3. 链上申购与链下基金真实申购之间的关系未完全定义

已识别问题：
- 真实基金仍是链下资产，链上申购后如何与链下发行商/管理人流程对接没有完全定稿。
- 当前讨论中至少存在两种解释：
  - 链上是真正的申购入口
  - 链上更像“意向申请 / 报名 / 预约”
- 投资人实名、链下确权、最终发行确认等动作没有统一建模。

影响：
- 影响前端文案
- 影响链上/链下状态同步
- 影响申购成功、分配成功、最终持有之间的业务含义

### 4. KYC / 合格投资者准入机制目前只有方向，没有完整流程

已识别问题：
- 链上地址本身不代表实名身份，无法直接满足基金申购的实名和适当性要求。
- 目前会议上提出的解决方向是白名单机制，但还没有完整流程定义。
- 仅凭当前 Ticket，仍不足以支撑“谁可以进入白名单、谁维护、何时导入、如何撤销”等细节。

当前已识别的方向：
- 使用白名单管理
- 仅通过 KYC 和合格投资者认证的用户可参与申购
- 支持 Excel 导入白名单

影响：
- 影响前台是否展示 `Subscribe`
- 影响 Admin / Operator 的后台需求
- 影响 demo 里是否需要 mock 合规流程

### 5. 分配机制存在“自动分配”与“线下分配后上传”两种模式

已识别问题：
- 当前 Ticket 默认更偏系统内分配，但会议中明确提出链下人工分配后上传结果也是重要场景。
- 对基金场景来说，发行方可能需要先在线下完成资格审查，再决定最终分配结果。
- 当前系统尚未明确是否支持两种模式并存。

影响：
- 影响 Issuance 流程中的 `Allocation rule`
- 影响后台功能与上传能力
- 影响链上分配步骤的前置数据来源

建议方向：
- 创建基金时允许选择：
  - 系统自动分配
  - 线下分配后上传结果

### 6. Redemption 流程目前只是简化版，未覆盖真实开放式基金约束

已识别问题：
- 赎回不是随时发生，很多基金会有锁定期、赎回窗口、公示期、单用户限额。
- 当前 demo 里已有 `Open For Redemption` 逻辑，但会议中明确要求：
  - 赎回前需提前一周公示
  - 部分基金只在特定时间窗口开放赎回
  - 可设置单用户最大赎回额度
- 当前 Ticket 对“列表选择、历史逻辑、额度限制、窗口机制”覆盖还不够。

影响：
- 影响 Redemption 状态机
- 影响前端字段设计
- 影响是否需要历史记录、待处理列表、额度控制

### 7. Distribution 的业务定义比当前 Ticket 更复杂

已识别问题：
- 分红不改变份额，只改变资金，这一点已经明确。
- 但围绕分红，仍有多项实现问题未被完整写入 Ticket：
  - 分红资金从哪里来
  - 是否需要单独的 payout account
  - 采用什么 ERC20 / 稳定币发放
  - 按哪个时间点锁定持有人
  - 分红金额是否允许后台筛除/调整个别用户

影响：
- 影响 Distribution Create 表单字段
- 影响链上 payout 方案
- 影响对账与资金归属

### 8. Distribution 的实际发钱方式有两种，当前 Ticket 尚未正式定稿

已识别问题：
- 会议中明确识别出两种发钱方式：
  - 项目方直接 `transfer` 给用户
  - 项目方先 `approve`，用户自己 `claim / pull`
- 当前 Ticket 虽已偏向 `Accept Distribution` 模式，但没有把两种模式的取舍写成正式决策。
- 核心差异在于 gas 成本由谁承担。

影响：
- 影响钱包签名点
- 影响用户体验
- 影响链上合约接口设计

### 9. 发行材料与合规材料的要求未固定

已识别问题：
- 会议明确提到不同地区的材料要求不同，例如香港与新加坡不同。
- 因此发行时不应要求一次性上传所有材料。
- 当前 Ticket 虽已支持后续补充，但未把“为什么允许开放上传”写成决策背景。

影响：
- 影响 `Fund Documents` 的设计原则
- 影响是否必须上传某些法定文件
- 影响未来对接律所或合规顾问时的补充需求

### 10. 现有参考 demo 可参考但不可直接交付

已识别问题：
- 现有参考 demo 为视频 demo，不是可运行 live demo。
- 即便视觉可参考，也无法支撑现场交互式演示。
- 因此本轮工作不能只依赖参考视频，必须做成可运行演示。

影响：
- 影响工作量判断
- 影响排期预估
- 影响 scope 取舍

## 二、需要 Research / 外部确认的事项

### A. 需要向需求方 / Supervisor / CCB 侧确认

1. 本轮 demo 目标基金类型到底是哪一种
- 封闭式基金
- 开放式基金
- ETF 类产品
- 是否要求同时覆盖多种类型

2. 本轮 demo 的最低交付范围是什么
- 是否只要求演示核心流程
- 是否必须支持 live demo
- 是否需要包含开放式基金专属结算逻辑

3. 申购的业务含义是什么
- 真正链上申购
- 链上预约 / 意向申请
- 链上申购后链下最终确认

4. 白名单 / KYC / 合格投资者认证在 demo 中要做到什么程度
- 完整流程
- 只做 mock
- 只做名单控制不做身份系统

5. Redemption 是否要求：
- 提前公示期
- 周期性赎回窗口
- 单用户最大赎回额度
- 历史记录与待处理列表

6. Distribution 是否要求：
- 直接派发
- 用户主动领取
- 两种模式都展示

### B. 需要产品 / 业务侧进一步研究

1. 开放式基金与 ETF 的关键差异
- 是否都需要支持
- 对应的状态机有何不同
- 确权、结算、交易时间规则如何不同

2. 开放式基金的净值结算机制
- 是否每日计算 NAV
- 是否 T+1 确权
- 是否需要参考现有 `djft` 结算能力

3. 货币基金 / money market fund 的上链逻辑
- 是否允许链上申购
- 是否需要特殊定价/结算模型

4. Distribution 的链上资金模型
- 使用何种稳定币
- 是否单独配置 payout account
- 是否需要预存资金或额度授权

5. 份额分配的合规模式
- 自动分配
- 人工审核后上传
- 两种模式如何影响前后端结构

6. 链上确权时间点的定义
- Record date 如何锁定
- 在 record date 前后发生转让/变更时如何处理

### C. 需要技术 / 合约侧研究

1. Distribution 采用 `transfer` 还是 `approve + claim`
- gas 成本分别由谁承担
- 用户体验差异
- 合约复杂度差异

2. Redemption 的合约模型
- 是否 burn 份额 token
- 是否转入 redemption contract
- 链上与链下回款如何对账

3. 白名单导入与校验机制
- Excel 导入只是 UI 入口还是实际写入链上/数据库
- 白名单是链上校验还是后端校验

4. 线下分配结果上传
- 上传格式
- 校验机制
- 如何与链上 allocation 对接

## 三、可直接进入开发 / 对齐的结论

### 1. 三条主流程方向不变

当前仍然可以围绕三条主流程推进：
- Issuance
- Redemption
- Distribution

### 2. Demo 范围应优先收敛

建议本轮 demo 以以下优先级推进：
- P1：封闭式基金主流程
- P1：开放式基金基础规则展示
- P2：赎回窗口与限制
- P2：分红方式展示
- 暂缓：ETF

### 3. 需要尽快补的文档

建议下一步补这三类文档：
- 生命周期范围说明
- 开放式基金 vs 封闭式基金规则对比
- Distribution 资金发放模型对比

### 4. 当前 Ticket 后续应重点增强的章节

建议重点补充：
- Fund type 差异化规则
- KYC / whitelist 模型
- Allocation 模式选择
- Redemption 窗口与额度限制
- Distribution 的 payout model

- Distribution 采用 `transfer` 还是 `approve + claim`
- 开放式基金是否必须支持 NAV / 每日结算
